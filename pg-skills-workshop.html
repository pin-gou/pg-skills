1. 为什么需要品构

让 AI 写出可托付的代码 · 让品控长在结构里

品构 = 品质 × 构造。
我们不"让 AI 写代码"——我们让 AI 在一个被 SSOT 钉死的结构里写代码：每一行都有出处，每一个阶段都有产物，每一个失败都有回放。
故名「品构」：可托付的代码，靠结构本身托付，而不是靠人盯。

1.1 为什么需要「可托付」

传统 AI 编码的真实代价不是"写得慢"——是写完不敢合。以下四种灾难在每个用过 AI 的团队身上都发生过：

✕ 让 AI 自由写

  • 没人敢合 master：AI 一锤定音 → 出问题归谁？
  • 故障再现：AI 改完就忘 → 上次怎么好的不知道
  • "在我机器上能跑"：AI 的测试环境 ≠ 线上
  • 漏 1 个就是事故：review 全靠人眼 → AI 不签自己的字

✓ 品构协创

  • 三件套追责：每段都有 proposal / design / tasks，按谱追责
  • 一键 replay：pipeline.events JSONL 全留痕，事件溯源
  • 环境归零：project.yaml SSOT + hook 协议，差异归零
  • 门控 + 签字：TDRVG 评分 ≥80 + final-gate 人工一行

1.2 品构 = 品质 × 结构

「品构」这个名字藏着全部设计哲学：

命名释义：质量靠结构本身托付——SSOT（project.yaml、execution-manifest.yaml）钉死"该有什么"，pipeline（stages × tracks × TDRVG）钉死"该怎么走"，每个 sub-agent 落一 checkbox 钉死"做没做完"。AI 在这套结构里写代码，写错了也是系统先看见。

1.3 托付的 4 个支柱

「可托付」不是空话——它由这 4 个具体可触达的支柱承载，对应 §1.1 的 4 种灾难一一化解：

#支柱化解的灾难机制
1 出生证明 没人敢合 master 每段变更必出 proposal.md + design.md + tasks.md + execution-manifest.yaml 4 件套，明确"做什么 / 怎么做 / 如何验 / 谁来做"；缺失 → workflow_failed 终止
2 全程留痕 故障再现 2-build/pipeline.events append-only JSONL 作为唯一持久化入口（事件溯源引擎）；replay 子命令重放比对，tasks.md checkbox 标注断点恢复位置
3 环境归零 在我机器上能跑 project.yaml 是项目级 SSOT；environments 段 + hook 协议（pg-invoke-hook.py）抹平机器差异，所有 role 启停走同一套 PG_* env vars
4 门控 + 签字 漏 1 个就是事故 TDRVG 五阶段（含可选 code review）+ pg-build/gate 5 维加权评分（gate_score ≥ 80）+ final-gate 人工一行签字；不及格走 fix-cycle / gate-fix-cycle / review-cycle 子 pipeline

这 4 个支柱就是后续章节的目录骨架：§3 讲 支柱 3（环境归零）、§4 讲 支柱 1+4（出生证明 + 门控）、§5 讲支柱 4 的 fix-issue 路径、§6 讲支柱 2 的回归闭环。

1.4 AI 在每一步的角色

「让 AI 写出可托付的代码」——这里「写出」二字要分开看：

流程AI 角色是否署名谁兜底
定界 /1-pg-define 半结构化"准出图员"：扫 repo、把模糊表达收成 in/out of scope AI 决策部分：辅助澄清；最终签名 = 人 人用 question 工具三选一
提案 /2-pg-propose 被规训的"出图员"：填 proposal / design / tasks 骨架 + 出 execution-manifest.yaml AI 全程写；产物必含 review-notes.md 人审 review-notes 圈 checkbox；调过才 apply
构建 /3-pg-build 受约束的"代码工人"：被 TDRVG + stages × tracks pipeline 约束的 sub-agent 派遣 每段 commit 写明 dispatch 来源（dispatch file） events_log 全留痕；replay 可重放；review cycle 独立计数
回归 /4-pg-regression 测试修缮工（带边界）：fix-test 只修 A/B 类（断言漂移 / 选择器过期），C 类（生产 bug）不碰 改的就 commit；不改的就拉问题清单走 fix-issue pg-check-fix-test-boundary.py 二次守门
修复 /5-pg-fix-issue 被复现命令约束的"诊断员"：先出复现方案 + success criteria，再改代码 Phase 3 用户签字；fix 失败 → escalate executor 验证 → PR 或 ESCALATE

「可托付」的内在承诺：每一段被 AI 写出的代码，都先定了界、有人审了图、跑过了门控、留了回放。"可托付"不是流程写在文档里——是流程对每个 commit 都收一次口。

2. 架构全景——三层模型

「让 AI 写出可托付的代码」—— 我们从结构开始建。一个项目对品构是 3 层：共享能力层 / 项目配置层 / 业务代码层。

2.1 三层架构

核心原则：L1 不含任何项目特定知识（无 webvirt、无 pangee）；L2 是项目的单一可信源；L3 只关心业务代码。

2.2 项目目录结构

.pg/
├── project.yaml                  # 统一配置 SSOT（六段结构）
├── changes/                      # pg-build caller 路由：变更提案与过程产物
│   ├── add-login-page/           # 某次变更
│   │   ├── proposal.md           # 做什么、为什么做
│   │   ├── design.md             # 怎么做、验证标准
│   │   ├── tasks.md              # stages × tracks  任务列表（含可选 review sub）
│   │   ├── execution-manifest.yaml  # 按 tasks.md 结构化生成的 pipeline 编排清单（runner 读取）；含 stages[i].environment 字段
│   │   └── 2-build/              # pipeline 过程产物（pipeline.events / 报告 / gate-score）
│   └── archive/                  # 已完成变更归档
├── regression/                   # pg-regression caller 路由：回归测试与问题清单
│   ├── frontend.json             # 生产代码问题清单（SSOT, 跨 run 累积）
│   ├── frontend--NN/       # 单次 run 目录（fix-issues / fix-test 留痕 + 报告）
│   └── ...
├── fix-issue/                    # pg-fix-issue caller 路由：bug 修复会话
│   └── fix--/  # 每次修复一个独立目录
│       └── dev-local/logs/       # hook 日志与产物
├── ad-hoc/                       # ad-hoc caller 路由：用户/agent 直接调 hook
│   └── auto--/        # 留空 --session 时自动生成
├── cronjobs/                     # 定时任务：crontab + opencode run 驱动批量任务
│   ├── run-opencode.sh           # 入口脚本（stash → master → opencode run → pop）
│   ├── prompt.txt                # 待执行提示词（git ignored）
│   └── cron.log                  # 定时执行日志
├── hooks/                        # 环境生命周期脚本（仅 environments 维度）
│   ├── role-backend-start.sh
│   ├── role-agent-start.sh
│   ├── env-dev-local-prepare.sh
│   └── ...
├── code-review/                  # pg-build v2.6 review phase SSOT（由 pg-init-project Phase 2.5 生成）
│   ├── code-review.yaml          # profile 索引（default + 语言 profile 的 Union）
│   ├── default/                  # 兜底 profile（含 design_alignment / scope_creep 等）
│   ├── java-spring/              # java/kotlin/scala 自动派发
│   ├── go/                       # go 自动派发
│   └── vue3/                     # ts/js/vue 自动派发
├── context/                      # 项目上下文快照（AGENTS.md 提取后缓存）
└── skills/                       # pg-skills 嵌入（git subtree），pg init 在 .opencode/ 创建 symlink

caller × session 双维度路由：上面 changes/ / regression/ / fix-issue/ / ad-hoc/ 四个目录由 PG_RUN_CALLER 决定一级目录，由 PG_RUN_SESSION 决定二级目录。runner 自动按调用方身份把日志、PID、result.json 写到对应路径，无需手动指定。cronjobs/ 不属于 hook 协议路由，由项目自管；典型场景是 crontab 触发 pg-regression 或 pg-propose 批量任务。

2.3 AGENTS.md 的角色

AGENTS.md 是 AI 的"入职手册"——包含项目的技术栈、包结构、数据库约定、编码风格、路径规范等。pg-propose 会自动从 AGENTS.md 提取上下文写入 .pg/context/summary.yaml。

3. 新项目接入——Two Pillars

「让 AI 写出可托付的代码」—— 我们先把骨架钉死在项目里。骨架对，AI 在里面写代码就是"合结构"的；骨架错，AI 写得越多越危险。

接入新项目，最重要的三件事：写对 project.yaml（含 review_level） + 写对 hooks + 写对 .pg/code-review/

3.1 四步接入

  1. git subtree 拉入 pg-skills（含 .pg/skills/examples/code-review/ 模板：default / java-spring / go / vue3 / security 五个 profile）
  2. pg init 生成骨架 + symlink
  3. 重启 opencode（加载 slash commands + skills + sub-agents，含新 pg-build/review / pg-build/fix-review）
  4. 在 opencode 中触发 pg-init-project skill（AI 自动扫描仓库，生成 project.yaml + hooks + .pg/code-review/）
# ① 用 git subtree 把 pg-skills 同步进项目
git remote add pg-skills git@gitee.com:shao_hq/pg-skills.git
git fetch pg-skills
git subtree add --prefix=.pg/skills pg-skills master --squash

# ② 跑 pg init 创建 .pg/ 骨架 + .opencode/ symlink
python3 .pg/skills/src/runtime/bin/pg init

# ③ 重启 opencode 后，在 opencode 中执行 pg-init-project
#    （Phase 1 扫仓库 → Phase 2 写 .pg/project.yaml（含 review_level）→ Phase 2.5 按 language 派发 review profile → Phase 3 写 .pg/hooks/ → Phase 4 doctor 校验）
# ④ 验证
python3 .pg/skills/src/runtime/bin/pg doctor
讲师强调：接入新项目最重要的三件事就是 ① 写对 project.yaml（骨架，含 review_level）、② 写对 hooks（血肉）、③ 写对 .pg/code-review/（review phase 的语言特异检查项源头）。这三件不对，AI 再怎么跑 pipeline 都是错的。

3.2 project.yaml —— 六段结构

modules 段

modules:
  backend:
    root: webvirt-backend
    language: java
    review_level: security          # TBD 复核；java/go/proto → security 推断；ts/py → standard；shell → none
    build: .pg/hooks/backend-build.sh     # 虽在 hooks/ 下，但不走 hook 协议
    lint: cd webvirt-backend && mvn checkstyle:check  # 也可能 inline
    test:
      unit: .pg/hooks/backend-test.sh
      integration:
        cmd: .pg/hooks/backend-test.sh
        timeout_seconds: 3600

review_level（pg-build review phase 开关，schema 默认 standard）：pg-init-project 按 language 推断初值——java/kotlin/scala/go/proto → security（含服务端核心/协议/凭据类代码），typescript/javascript/python → standard，shell → none，其他 → standard。该字段控制 review 阶段是否进入该 module 的静态审查；如需 opt-in security profile，配合 tracks.<id>.code_review_profiles: [security, ...] 使用。

environments 段

environments:
  dev-local:
    description: "本机全栈开发"
    prepare_env: { script: .pg/hooks/env-dev-local-prepare.sh, timeout_seconds: 300 }
    clean_env:   { script: .pg/hooks/env-dev-local-clean.sh, timeout_seconds: 120 }
    roles:
      backend:
        instances:
          - { name: backend-1, host: localhost, port: 9080 }
        actions:
          start: { script: .pg/hooks/role-backend-start.sh, timeout_seconds: 300 }
          stop:  { script: .pg/hooks/role-backend-stop.sh, timeout_seconds: 30 }
          logs:  { script: .pg/hooks/role-backend-logs.sh, args: ["{lines:100}"] }

tracks + stages 段

# tracks：定义在哪些 modules 上跑 TDRVG 循环（含 review 子阶段，由 pg-build v2.6 派发）
tracks:
  backend:
    modules: [backend, agent-proto]
    max_fix_retries: 5
    code_review_enabled: true
    code_review_profiles: [security]   # 鉴权/协议/凭据类变更，手动 opt-in
  agent:
    modules: [agent, agent-proto]
    code_review_enabled: true
  frontend:
    modules: [frontend]
  proto-gen:
    type: simple                       # simple track 自动 code_review_enabled=false

# stages：定义 pipeline 阶段编排（on_conditions 决定 change 是否进入该 stage）
stages:
  - name: prepare-env-scripts
    tracks: [backend, agent, frontend]
    test_key: unit
    on_conditions:                       # 仅当改动命中 .pg/hooks/ 才启用
      - any_change_in: ["\\.pg/hooks/.*\\.sh$"]
  - name: dev
    tracks: [backend, agent, openapi-gen, frontend]
    test_key: unit
    gate: all_pass
    environment:
      required: true
tracks 字段速查：
  • modules：本 track 关联的 module id 列表（runner 用此定位 build/lint/test 命令）
  • max_fix_retries：verify→fix 子 pipeline 的总次数上限（默认 5）
  • type：standard（默认，TDRVG 四阶段 + 可选 review）/simple（commands 列表直跑，自动跳过 review）
  • code_review_enabled：是否派发 pg-build/review agent。propose 阶段由 pg-gen-tasks-skeleton.py 按 track.code_review_enabled 决定 tasks.md 含 4 还是 5 sub（不含 review 即 silent skip）
  • code_review_profiles：可选。手指定派的 profile 列表，按顺序合并（Union）；常见的 default 兜底 + security opt-in；语言自动派发的 profile（java→java-spring / go→go / ts→vue3）由 pg-build 直接读 .pg/code-review.yaml，不需要在这里配置

3.3 Hook 协议详解

两条铁律

维度走 hook 协议？配置位置调度方式
Environments
prepare_env / clean_env / role start/stop/logs		 ✅ 走 hook .pg/hooks/<name>.sh ./pg-run（用户）/ pg-invoke-hook.py（agent）统一入口
Modules
build / lint / test		 ❌ 不走 hook project.yaml modules.<m>.{build,lint,test} Runner 直接 timeout N bash -c '<cmd>'
⚠ 常见误区：module 命令虽然可以是 .pg/hooks/backend-build.sh 这样指向脚本文件，但它是被 runner 当作普通 shell 命令执行的——没有 PG_* 注入，不走 hook 生命周期。是否走 hook 协议取决于配置在 project.yaml 的哪个位置，不在于是不是脚本文件。

Hook 标准模板

#!/usr/bin/env bash
# .pg/hooks/role-backend-start.sh
set -uo pipefail

# === 1. 加载 hook-helpers（必须）===
source "$PG_SKILLS_PATH/src/runtime/lib/hook-helpers.sh"
trap 'pg_fail_on_error $? $LINENO' ERR

# === 2. 业务逻辑 ===
start_backend() { ... }

# === 3. 报告结果 ===
DURATION=$(($(date +%s) - START))
pg_exit --status=pass --duration=$DURATION

# 失败时用 pg_fail：
# pg_fail --category=test_failure --code=PG-E-0601 \
#         --message="mvn test failed" --agent-recoverable=true

注入的环境变量（v5 协议）

变量说明适用范围
PG_SKILLS_PATHpg-skills 根路径全部
PG_PROJECT_ROOT项目根路径全部
PG_RUN_CALLER调用方身份（pg-build / pg-regression / pg-fix-issue / pg-agent / ad-hoc）全部
PG_RUN_SESSIONsession 名（与 caller 正交）全部
PG_HOOK_LOG_DIRhook 派生的业务日志目录（预拼绝对路径）全部
PG_LOG_FILEhook 自身 stdout/stderr 目标路径全部
PG_RESULT_FILE写 result.json 的路径全部
PG_STAGE当前 stage 名全部
PG_ENV当前 environment（dev-local / dev-3tier）全部
PG_ROLErole 名role action 时
PG_INSTANCE_NAMEinstance 名role action 时
PG_INSTANCE_HOSTinstance hostrole action 时

v4 之前曾用 PG_SKILL_NAME / PG_CHANGE_NAME，现已统一为 PG_RUN_CALLER / PG_RUN_SESSION。模块维度（build / lint / test）不走 hook 协议，PG_MODULE / PG_MODULE_ROOT 已被移除。

两套入口：用户用 ./pg-run，agent/skill 用 pg-invoke-hook.py

hooks 操作有两套入口，使用场景不同：

  • 用户手工操作：./pg-run — 交互式菜单，无需记参数
  • Agent / SKILL 编排流：pg-invoke-hook.py — 底层 CLI，传入完整参数

pg-invoke-hook.py 是底层 CLI，LLM 和 runner 不直接调用 hook 脚本，通过它统一调度：

# Agent / SKILL 内部调用（用户请用 ./pg-run）
python3 .pg/skills/src/runtime/bin/pg-invoke-hook.py invoke-hook \
  --session <session-name> \
  --env dev-local \
  --role backend \
  --instance backend-1 \
  --action start \
  --caller pg-build

# 用户手工查看日志
./pg-run --env dev-local --role backend --instance backend-1 --action logs --tail-lines 20

# 查询环境准备状态（agent 用）
python3 .pg/skills/src/runtime/bin/pg-invoke-hook.py status \
  --session <session-name>

3.5 新项目接入 Checklist

接入流程的前 3 步（git subtree → pg init → pg-init-project）会自动生成 project.yaml、hooks 骨架、.pg/code-review/（Phase 2.5 按 language 派发）。但 AI 推断可能与实际项目有偏差，需要人工核对产物准确性，并通过 ./pg-run 对所有 hooks 做端到端测试验证。

  • git subtree 拉入 pg-skills → .pg/skills/（含 examples/code-review/ 模板目录）
  • pg init 生成骨架 + symlink
  • 在 opencode 中触发 pg-init-project skill（自动生成 project.yaml + hooks 模板 + .pg/code-review/）
  • 人工核对自动产物
  • modules：每个模块的根目录 + build/lint/test 命令是否准确
  • modules..review_level 是否符合实际敏感度（java/go/proto 推断 security；ts/py → standard；shell → none）
  • environments：环境列表、role 列表、actions 是否完整
  • tracks / stages：编排是否符合项目结构；tracks.<m>.code_review_enabled 是否需要为鉴权/协议模块开启
  • fix_issue / regression：参数是否合理
  • 每个 role 的 start / stop / logs 脚本：业务命令是否准确（CMD_PLACEHOLDER 是否替换）
  • 每个 env 的 prepare_env / clean_env 脚本：环境准备/清理逻辑是否对
  • lib/common.sh SSOT：是否正确复制到 .pg/hooks/lib/
  • .pg/code-review/ 是否按 language 派发（java → java-spring, go → go, ts → vue3）；security profile 未自动拷（opt-in）
  • （可选）声明 actions.health_check → pg-init-project 自动生成 health_check hook
  • 通过 pg-run 验证 hooks
  • pg doctor 校验（含 hooks/lib/common.sh 检查）
  • 人工 grep 验证 .pg/code-review/<language>/ 检查项细则文件存在（doctor 不覆盖）
  • ./pg-run --module <m> --action build/lint/test.<key> 验证 module 命令直跑
  • ./pg-run --env <env> --role <role> --instance <i> --action logs 验证 hook 可加载（只读）
  • 菜单 "准备环境并启动所有实例" → "停止所有实例并清理环境" 一键验证所有角色 + 环境级 hook（便捷）
  • 或单独验证：./pg-run --env <env> --role <role> --instance <i> --action start/stop + ./pg-run --env <env> --action prepare_env / clean_env
关键提醒：project.yaml 的 modules.<m>.build 字段可以直接指向 .pg/hooks/ 下的脚本——比如 backend-build.sh 虽然是一个脚本文件，但它不在 hook 协议中，而是被 runner 当作普通 shell 命令执行。不要被"脚本在 hooks 目录下"误导，是否走 hook 协议取决于它在 project.yaml 的哪个位置。

本章结构：4.1 全景路线图 → 4.2 定界 → 4.3 提案 → 4.4 构建。每节都讲核心机制 + 实操要点。

4.2 定界 —— `/1-pg-define`

第一个问题：需求有多复杂？

关于"拆"的建议

比如"优化创建 VM 的性能" → 拆成 (1) 模板磁盘预缓存、(2) 异步状态推送、(3) 前端进度条展示，三个 change 分三周推进。每个 change 独立 propose → build → merge，互不阻塞，且每个都能在 LLM 上下文窗口内完成。

第三个问题：scope 想清楚了吗？

/1-pg-define 是一个探索姿态，AI 不是机械提问而是思考伙伴。它会帮你做这些事：

  1. 澄清模糊表述 —— "太慢是多慢？具体哪个场景？"把模糊词转化为可测量描述
  2. 划定边界 —— 影响范围是单一模块还是跨多个组件？明确 in / out of scope
  3. 暴露隐式假设 —— 挑战"必须用 X"的前提，打开替代方案空间
  4. 识别集成点 —— 会触及哪些现有功能？依赖其他系统或团队吗？
  5. 对标既有模式 —— grep 同类功能，遵循项目既有风格
  6. 揭示风险 —— 破坏性变更？兼容性？边界情况（并发、权限、空值）？
  7. 收敛完成标准 —— 怎么算做好了？区分 MVP 和迭代

⚠ 探索中的禁忌（自检信号）

  • 🔴 过早解决：跳过对问题的理解就"直接用 X 就行"
  • 🔴 功能蔓延：用户每说一个想法就延伸 3 个额外功能 —— 记住 YAGNI
  • 🔴 过度工程化：设计一个通用框架处理所有场景
  • 🔴 确认偏误：只在代码库中找支持自己假设的证据
  • 🔴 写代码冲动：发现自己在想"那行代码怎么写"而非"方案是否合理"

4.3 提案 —— `/2-pg-propose`

"边界有了，现在出图纸。图纸分四份人类可读文档 + 两份机器 SSOT，每份给不同的对象看。"

v0.7 变更点：v1 的 environment.yaml 已弃用，per-change 的环境选择写入 execution-manifest.yaml.stages[i].environment。manifest 是 5 类产物的唯一机器 SSOT，由 pg-gen-manifest.py 基于 tasks.md 结构化生成，LLM 不直接写。

核心机制：tasks.md 的 stages × tracks 展开

pg-propose 通过 pg-gen-tasks-skeleton.py 机械生成 tasks.md 的章节标题骨架，LLM 只负责填充 body 内容。整体分两步过滤：

Step 1：on_conditions 判定（仅入选中的 stage）

检查本 change 的 affected_paths 是否命中该 stage 的 on_conditions 规则：

  • 未命中 → 该 stage 整体跳过，tasks.md 中不生成任何章节，也不占章节号 N
  • 命中 → 进入 Step 2

Step 2：affected_tracks × sub 判定（含可选 review sub）

对 stage.tracks 列表中的每个 track：

  • simple track → 1 个 heading（cmd，按顺序直跑）
  • standard track + code_review_enabled=false → 4 个 heading：T→D→V→G
  • standard track + code_review_enabled=true → 5 个 heading：T→D→R→V→G（review 插在 dev 与 verify 之间，由 pg-build/review agent 执行静态审查）
  • 不在 affected_tracks 中 → 生成占位章节 - 无

章节号 N 是动态的：脚本 pg-gen-tasks-skeleton.py 给每个章节分配的 N 仅在当次生成时稳定（同一份 tasks.md 内），但跨 change 不一致（同一 track 在不同 change 的 N 可能不同）。下游消费方必须基于 sections JSON 的 N 值填，禁止硬编码。例：dev stage 的 backend track 章节号在 change A 是 2-4，在 change B 可能因为多走了 prepare-env-scripts stage 变成 3-5。

environment 选择：写入 execution-manifest.yaml

对于 environment.required: true 的 stage，pg-propose 会读取该 stage 的 selection_rules，根据本 change 的 affected_tracks 选定 environment 并写入 execution-manifest.yaml 的 stages[i].environment 字段（v0.7 起取代旧 environment.yaml）：

# .pg/changes/<change>/execution-manifest.yaml（节选）
stages:
  - name: dev
    environment:
      name: dev-local          # runner 直接从此字段读取，不再读 environment.yaml
      required: true
    tracks: [backend, agent, frontend, proto-gen]
场景选择依据
只改 backend + frontenddev-local单机能跑，无需跨主机
改 agent 且需要双 agent 通信dev-3tier跨主机拓扑，验证 libvirt 连接
默认dev-local评审阶段可升级到 dev-3tier

⚠ environment 选错代价很大

  • runner 在错误的 host 上启服务 → 启动失败
  • 用错的 hook 脚本 → 端口冲突或环境变量缺失
  • 跨机通信走错网段 → 联调失败

execution-manifest.yaml.stages[i].environment 缺失 → workflow_failed 硬性终止。这是 SSOT 的硬约束：runner 不知道该用哪个环境，宁可终止也不猜。v1 遗留的 environment.yaml pg-build v2 不再读取。

评审闭环：review-notes.md + /2.1-pg-propose-refine

  1. pg-propose 自审生成 review-notes.md（6 类审查项 + checkbox）
  2. 人工评审，把发现的问题在 review-notes 中标 [x] 或 [~]
  3. 执行 /2.1-pg-propose-refine，AI 按 scope 精准修改 proposal / design / tasks
  4. 通常需要 1-2 轮 refine 迭代，直到 review-notes 全部 [x]

人工 review 的硬规则：API 契约变更、DB schema 变更、跨模块能力演进，必须经人工 review 后才能进入 build 阶段。

4.4 构建 —— `/3-pg-build`（基于事件溯源引擎）

"图纸审完了，Runner 开始施工。人只做一件事：执行命令等结果。"

全景鸟瞰图：多 stage × 多 track 一次看完

整图三段叙事：① 顶部是编排器每轮执行的 4 步主循环（next → action → 派 sub-agent → record），是整张图的节奏；② 中部是 stage 横排、track 纵列的主 pipeline，场地，每个 track 内部走 TDRVG，失败时由虚线泡泡接住（fix / gate-fix / review 子 pipeline）；③ 右上是 final-gate + archive 终点。下面的每个小节就是这三段的局部放大版。

驱动流程：调用-派送-记录 循环

对应鸟瞰图的位置：这 4 个节点就是鸟瞰图顶部条带①的展开版——main loop 的每轮 4 拍。区别在于：本图加了 runner 输出的 action 取值（dispatch / phase_result / done / workflow_failed），告知编排器下一拍该派谁、该记录什么、该继续还是终止。

Runner 自动管的事（你不用操心）

  • 切到 feat/pg/<change> feature branch
  • 启动时 init commit（chore(<change>): bootstrap pg-build）
  • 每个 sub 阶段 record commit（chore(<change>): auto-record <item>:<sub> <status>）
  • 报告顺序编号 {track.id}-{N}-{kind}.md
  • final-gate pass 后自动归档到 changes/archive/

execution-manifest.yaml —— runner 的执行蓝图

在 runner 启动编排前，tasks.md（人类可读 markdown）需要先被结构化成 execution-manifest.yaml，这是 runner 直接消费的机器可读形态：

字段含义示例
stagesstage 列表（顺序）[prepare-env-scripts, dev, real-integration]
stages[].namestage 名（与 project.yaml.stages 对齐）dev
stages[].tracks本 stage 包含的 track 列表（顺序执行）[backend, agent, openapi-gen, frontend]
stages[].environment.name本 stage 选用的 environment 名dev-local / dev-3tier
stages[].test_key本 stage 跑哪个 test_keyunit / integration / e2e
tracks[].idtrack 名backend
tracks[].modules本 track 涉及的 modules（用于定位 build/lint/test 命令）[backend, agent-proto]
tracks[].typetrack 类型standard（默认 TDRVG 4-5 sub）/ simple（commands 列表直跑，自动跳过 review）
tracks[].code_review_enabled是否派发 review sub-agent；propose 决定 tasks.md 含 review sub；missing 时 silent skiptrue / false
tracks[].phase_prompts对应 4-5 sub 的 prompt 模板（test/dev/verify/gate；含 review 时多 review）由 pg-gen-manifest.py 派生

为什么需要 manifest？tasks.md 是带 checkbox 的人类文档（LLM 友好），但 runner 是程序（机器友好）。manifest 把 tasks.md 的 stages × tracks 二维结构扁平化成确定性数据结构：每个 stage 的 environment、test_key、track 顺序都冻结到 manifest，避免 runner 在执行时再次解析 markdown 带来的不确定性。manifest 是 disposable（可重新生成），但作为执行期 SSOT 必须存在。

manifest 由 pg-propose 在生成 tasks.md 后自动产出（LLM 不直接写，通过 CLI 工具基于 tasks.md 生成）。runner 启动时一次性读取，决定：

  • 要执行哪些 stage、顺序如何
  • 每个 stage 切到哪个 environment（优先于 project.yaml.stages[*].environment.name）
  • 每个 stage 跑哪个 test_key
  • 每个 track 派遣哪个 sub-agent（test / dev / verify / gate / fix / fix-gate / simple）

TDRVG 阶段：每个 track 内部循环（pg-build v2.6）

每个 standard track 内部按顺序执行 4 或 5 个 sub——多了 review（静态代码审查，dev 与 verify 之间）：

  • 4 sub（默认）：test → dev → verify → gate（track 未开启 review 时）
  • 5 sub（review 开启）：test → dev → review → verify → gate（code_review_enabled=true 时）

review 阶段是干嘛的：在 verify（集成/E2E）之前先做一轮静态代码审查，目的是拦截"实现与设计不一致 / scope creep / 模式不一致 / 测试契约弱"等静态代码问题，降低后续 fix cycle 成本。它与 verify 的区别：

  • review（静态）：design 对齐、scope creep、模式一致、文件位置、测试契约；不改代码
  • verify（运行时）：V-* 验证项；启动服务跑测试
  • 触发 fix 的失败项不同：review 的 R-*（设计/模式）由 pg-build/fix-review agent 修；verify 的 V-*（功能/集成）由 pg-build/fix agent 修。两个 fix 循环独立计数，互不干扰。

环境准备与清理

当 stage 的 environment.required: true 时，runner 在进入该 stage 前自动执行 prepare_env，完成后自动 clean_env。对应鸟瞰图中部的橙色 env 边界线（在 Stage A ↔ Stage B 之间）：从 A 离开时跑 clean_env，进 B 时跑 prepare_env，整条线由 runner 检测 stage 切换自动插入，无需 LLM 参与。

使用哪个环境？由 execution-manifest.yaml.stages[i].environment 字段决定（pg-propose 阶段已选定）。runner 通过 pg-invoke-hook.py invoke-hook --action prepare_env 触发，不传 --role / --instance（env-level 操作）。

幂等性：prepare_env 是幂等的——多次执行不会产生副作用。hook 脚本内已做"已就绪则跳过"的判断。

流程中断怎么恢复

中断场景恢复方式
会话中断（网络/超时） 重新 pg-pipeline-runner.py next，runner 从断点恢复，不丢失进度
Sub-agent 返回 result.json 缺失 编排器自动重试一次（v2.4 保护流程），仍缺失 → fatal result_json_missing_after_retry；或改用 v2.5 的 --result-json <path> 显式传路径，7 字段自动从文件加载
Sub-agent 返回空 pg-pipeline-runner.py record failed 触发重试（verify→fix 循环最多 max_fix_retries 次，默认 5）
workflow_failed（铁律） 人工介入：修复根因 → 重新 pg-pipeline-runner.py next → runner 读取 2-build/pipeline.events 重放 + 扫描 tasks.md checkbox，从第一个含未完成任务 [ ] 的章节恢复
回归比对 pg-pipeline-runner.py replay <change> 基于 event log 重放并比对结果，用于排查不一致

⚠ workflow_failed 恢复的关键点

不是从头开始！runner 从 pipeline.events 重放事件流，并扫描 tasks.md checkbox，定位到第一个未勾选 [ ] 的章节作为恢复起点。已完成的 [x] 章节全部跳过。这就是为什么 propose 阶段每个子任务都必须有 checkbox —— 它是 build 阶段断点恢复的依据。

4.4.5 simple track —— 跳过 sub-agent 的轻量编排

并非所有 track 都需要完整 TDRVG 四 / 五阶段。对于纯代码生成、文档同步、模板替换这类无设计决策的小活，可以用 simple 类型的 track：

simple track 配置示例

tracks:
  openapi-gen:
    type: simple                       # 关键：跳过 sub-agent；manifest 不含 phases.review
    modules: [backend]
    commands:
      - cd webvirt-backend && mvn -pl webvirt-openapi openapi:generate
      - cp webvirt-backend/target/openapi.json webvirt-frontend/src/api/
维度standard tracksimple track
编排模型TDRVG 4-5 阶段 sub-agent（T/D/optional-R/V/G）commands 列表直跑
review phase由 code_review_enabled 决定是否派发 pg-build/reviewmanifest 不生成 review phase → 自动 skip
重试策略max_fix_retries（verify→fix 循环）；review 走 max_review_fix_retries（独立）按每条命令的 on_failure（默认 workflow_failed）
报告产物{track}-N-{review,fix-review,verify,verify-fix,gate,gate-fix}.md + gate-assessment.mdrunner 自动写 {track}-N-simple.md（stdout/stderr 摘录）
适用场景跨模块、需设计评审、需静态/运行时审查openapi-gen、license header 注入、模板替换、单测命名规范化等

选择原则：如果 track 不需要"设计决策 → 代码 → 验证 → 门控"四步走、且失败信号可直接从命令 exit code 读出，就用 simple。否则用 standard。simple track 无 review 阶段（不在 language 自动派发范围内），适合纯机械脚本。

4.4.6 gate-score —— 5 维加权评分

每个 track 在进入 gate 阶段时，由 pg-build/gate sub-agent 按 5 个维度打分，总分 ≥ 80 才视为 gate 通过。相比传统的 bool 通过/失败，gate-score 能区分"差一点点"和"差很多"：

维度权重评估内容
correctness35%行为是否符合 proposal.md 描述、单元测试覆盖关键路径
security25%权限校验、敏感字段脱敏、依赖 CVE 扫描
maintainability15%命名规范、模块边界、日志规范、复用度
performance15%关键路径延迟、数据库查询 N+1、资源占用
spec_coverage10%design.md 验收点是否全部被测试或文档覆盖

评分如何驱动 fix-gate 循环

  1. gate agent 输出结构化 JSON（summary 字段必含 gate_score: <0-100> + p0_failures: [...]）：{"scores": {...}, "total": 86, "threshold": 80, "passed": true, "weak_dims": []}
  2. total ≥ 80 → gate pass，pipeline 推进
  3. total < 80 → 进入 fix-gate 子 pipeline（fix-gate → verify → gate），runner 把 weak_dims 传给 pg-build/fix-gate sub-agent 定向修复
  4. fix-gate 循环耗尽 max_fix_retries（v2.3 起 verify→fix 与 gate→fix-gate 共用同一计数器）→ final-gate 触发人工评审

前置门控：gate-assessment.md

final-gate 启动前必须存在 2-build/gate-assessment.md，汇总所有 track 的 gate-score + 弱项清单。缺失则 final-gate 硬阻断，避免"未评审就合入"。

⚠ 分数驱动 ≠ 形式主义

gate-score 不是为了让 AI "凑分数"。权重和阈值在 pg-skills 上游 SSOT 定义，项目内不可随意调。如发现某维度评分与人类判断长期偏离，应反馈到上游调整 prompt 模板，而非本地 hack。

Q&A

Build 跑到一半失败了怎么办？

查看 .pg/changes/<name>/2-build/*.md 中的报告。gate 或 verify 报告会写明失败原因。修复后重跑 /3-pg-build。

怎么知道当前 pipeline 跑到哪一步了？

查看 .pg/changes/<name>/2-build/pipeline.events（append-only JSONL），或调 pg-pipeline-runner.py progress <change> 看人类可读摘要。

提案评审时发现了问题，怎么改？

在 review-notes.md 中标记 [ ] 为 [x]（已修）或 [~]（跳过），然后执行 /2.1-pg-propose-refine。AI 会自动按 scope 精准修改 proposal / design / tasks。

怎么新增一个回归测试套件？

编辑 .pg/project.yaml 的 regression.suite 段，增加一个新 suite 配置：

regression:
  suite:
    my-new-suite:
      environment:
        name: dev-local
        required_roles: []
      module: backend
      test_keys: [unit]
怎么加一个测试环境（比如 staging）？

编辑 .pg/project.yaml 的 environments 段，新增一个环境 + 各 role 的 actions。然后为该环境编写对应的 hook 脚本（prepare_env / clean_env / role start/stop/logs）。

pg-skills 版本怎么升级？
# 升级到最新版
pg upgrade

# 升级到指定版
pg upgrade v0.6.0

# 查看远程可用版本
pg upgrade --list

# 校验
pg doctor
怎么验证 hooks 写得对不对？

用 ./pg-run 的 logs 动作（只读操作，安全无害）验证 hook 脚本是否能正常加载并执行：

./pg-run --env dev-local --role backend --instance backend-1 --action logs --tail-lines 10
module 命令和 hook 脚本的区别到底是什么？

核心区别两点：

  1. 是否有 PG_* 注入：hook 脚本有 $PG_ROLE、$PG_ENV 等；module 命令没有
  2. 错误处理方式：hook 脚本用 pg_exit / pg_fail 写 result.json；module 命令只看 exit code

判断标准很简单：看它在 project.yaml 中配置在哪个位置——modules 下的都是 module 命令，environments 下的都是 hook。
