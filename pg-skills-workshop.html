1. 为什么需要 pg-skills

AI 辅助开发的标准化持续迭代工作流

1.1 常见痛点

✕ 传统 AI 编码

  • 需求模糊 → AI 猜着做 → 反复返工
  • 范围蔓延 → 做了不该做的事 → 破坏现有功能
  • 质量不可控 → "看起来对但测试挂了"
  • 环境差异 → "在我机器上能跑"

✓ pg-skills 方式

  • 先定界再提案，需求对齐再编码
  • pipeline 自动编排，绝不越界
  • TDVG 四阶段门控，不合格不通过
  • 统一 project.yaml SSOT，环境单一入口

1.2 核心三句话

🔵 定界（Define）：改之前先划清边界，AI 不猜、不越界
🟢 提案 → 构建（Propose → Build）：先生成设计文档，人审完了再编码
🟠 回归 → 修复（Regression → Fix）：测试是质量防线，问题统一管理闭环

1.3 三个闭环对比

维度普通 AI 编码pg-skills
开始前自然语言描述/2-pg-propose 生成 proposal + design
编码中AI 自由发挥Pipeline 按 stages × tracks 编排，不可越界
完成后用户自己跑测试自动 TDVG + Gate 审查
出问题人工定位+猜测/5-pg-fix-issue 系统化诊断修复

2. 架构全景——三层模型

2.1 三层架构

核心原则：L1 不含任何项目特定知识（无 webvirt、无 pangee）；L2 是项目的单一可信源；L3 只关心业务代码。

2.2 项目目录结构

.pg/
├── project.yaml           # 统一配置 SSOT（六段结构）
├── changes/               # 变更产物
│   ├── add-login-page/    # 某次变更
│   │   ├── proposal.md    # 做什么、为什么做
│   │   ├── design.md      # 怎么做、验证标准
│   │   ├── tasks.md       # stages × tracks  任务列表
│   │   └── 2-build/       # pipeline 过程产物
│   └── archive/           # 已完成变更归档
├── hooks/                 # 环境生命周期脚本
│   ├── role-backend-start.sh
│   ├── role-agent-start.sh
│   ├── env-dev-local-prepare.sh
│   └── ...
├── context/               # 项目上下文快照
└── skills/                # pg-skills 嵌入（git subtree）

2.3 AGENTS.md 的角色

AGENTS.md 是 AI 的"入职手册"——包含项目的技术栈、包结构、数据库约定、编码风格、路径规范等。pg-propose 会自动从 AGENTS.md 提取上下文写入 .pg/context/summary.yaml。

3. 新项目接入——Two Pillars

接入新项目，最重要的两件事：写对 project.yaml + 写对 hooks

3.1 4 步接入流程

  1. git subtree 拉入 pg-skills
  2. pg init 生成骨架 + symlink
  3. 重启 opencode（加载 slash commands + skills + sub-agents）
  4. 在 opencode 中触发 pg-init-project skill（AI 自动扫描仓库，生成 project.yaml）
# ① 用 git subtree 把 pg-skills 同步进项目
git remote add pg-skills git@gitee.com:shao_hq/pg-skills.git
git fetch pg-skills
git subtree add --prefix=.pg/skills pg-skills master --squash

# ② 跑 pg init 创建 .pg/ 骨架 + .opencode/ symlink
python3 .pg/skills/src/runtime/bin/pg init

# ③ 重启 opencode 后，在 opencode 中执行 pg-init-project
# ④ 验证
python3 .pg/skills/src/runtime/bin/pg doctor
讲师强调：接入新项目最重要的两件事就是 ① 写对 project.yaml（骨架）和 ② 写对 hooks（血肉）。这两件不对，AI 再怎么跑 pipeline 都是错的。

3.2 project.yaml —— 六段结构

modules 段

modules:
  backend:
    root: webvirt-backend
    language: java
    build: .pg/hooks/backend-build.sh     # 虽在 hooks/ 下，但不走 hook 协议
    lint: cd webvirt-backend && mvn checkstyle:check  # 也可能 inline
    test:
      unit: .pg/hooks/backend-test.sh
      integration:
        cmd: .pg/hooks/backend-test.sh
        timeout_seconds: 3600

environments 段

environments:
  dev-local:
    description: "本机全栈开发"
    prepare_env: { script: .pg/hooks/env-dev-local-prepare.sh, timeout_seconds: 300 }
    clean_env:   { script: .pg/hooks/env-dev-local-clean.sh, timeout_seconds: 120 }
    roles:
      backend:
        instances:
          - { name: backend-1, host: localhost, port: 9080 }
        actions:
          start: { script: .pg/hooks/role-backend-start.sh, timeout_seconds: 300 }
          stop:  { script: .pg/hooks/role-backend-stop.sh, timeout_seconds: 30 }
          logs:  { script: .pg/hooks/role-backend-logs.sh, args: ["{lines:100}"] }

tracks + stages 段

# tracks：定义在哪些 modules 上跑 TDV 循环
tracks:
  backend:
    modules: [backend, agent-proto]
    max_fail_retries: 3
    max_gate_fix_retries: 2
    review_level: security
  agent:
    modules: [agent, agent-proto]
  frontend:
    modules: [frontend]

# stages：定义 pipeline 阶段编排
stages:
  - name: dev
    tracks: [backend, agent, openapi-gen, frontend]
    test_key: unit
    gate: all_pass
    environment:
      required: true

3.3 Hook 协议详解

两条铁律

维度走 hook 协议？配置位置调度方式
Environments
prepare_env / clean_env / role start/stop/logs		 ✅ 走 hook .pg/hooks/<name>.sh pg-invoke-hook.py 统一入口
Modules
build / lint / test		 ❌ 不走 hook project.yaml modules.<m>.{build,lint,test} Runner 直接 timeout N bash -c '<cmd>'
⚠ 常见误区：module 命令虽然可以是 .pg/hooks/backend-build.sh 这样指向脚本文件，但它是被 runner 当作普通 shell 命令执行的——没有 PG_* 注入，不走 hook 生命周期。是否走 hook 协议取决于配置在 project.yaml 的哪个位置，不在于是不是脚本文件。

Hook 标准模板

#!/usr/bin/env bash
# .pg/hooks/role-backend-start.sh
set -uo pipefail

# === 1. 加载 hook-helpers（必须）===
source "$PG_SKILLS_PATH/src/runtime/lib/hook-helpers.sh"
trap 'pg_fail_on_error $? $LINENO' ERR

# === 2. 业务逻辑 ===
start_backend() { ... }

# === 3. 报告结果 ===
DURATION=$(($(date +%s) - START))
pg_exit --status=pass --duration=$DURATION

# 失败时用 pg_fail：
# pg_fail --category=test_failure --code=PG-E-0601 \
#         --message="mvn test failed" --agent-recoverable=true

注入的环境变量

变量说明适用范围
PG_SKILLS_PATHpg-skills 根路径全部
PG_PROJECT_ROOT项目根路径全部
PG_RESULT_FILE写 result.json 的路径全部
PG_LOG_FILE写 stdout/stderr 的路径全部
PG_CHANGE_NAME当前 change 名全部
PG_STAGE当前 stage 名全部
PG_ENV当前环境（dev-local / dev-3tier）全部
PG_ROLErole 名role action 时
PG_INSTANCE_NAMEinstance 名role action 时
PG_INSTANCE_HOSTinstance hostrole action 时

pg-invoke-hook.py 统一入口

LLM 和 runner 不直接调用 hook 脚本，通过 pg-invoke-hook.py 统一调度。

# 调用 role action
python3 .pg/skills/src/runtime/bin/pg-invoke-hook.py invoke-hook \
  --change <change-name> \
  --env dev-local \
  --role backend \
  --instance backend-1 \
  --action start

# 查看日志（不启动服务，安全无害）
python3 .pg/skills/src/runtime/bin/pg-invoke-hook.py invoke-hook \
  --change demo \
  --env dev-local \
  --role backend \
  --instance backend-1 \
  --action logs \
  --tail-lines 20

# 查询环境准备状态
python3 .pg/skills/src/runtime/bin/pg-invoke-hook.py status \
  --change <change-name>

讲师提问：为什么不直接用 bash 跑 hook 脚本，而要用 pg-invoke-hook.py？
答案：因为 hook 协议需要注入 PG_* 环境变量、写 result.json、标准化错误分类。直接跑脚本会丢失这些上下文。

3.5 新项目接入 Checklist

  • git subtree 拉入 pg-skills → .pg/skills/
  • pg init 生成骨架 + symlink
  • 写 project.yaml（最重要）
  • modules：每个模块的根目录 + build/lint/test 命令
  • environments：有哪些环境，每个环境有哪些 role
  • tracks：哪些 track，每个 track 跑哪些 module
  • stages：分几个阶段，阶段内 track 顺序
  • fix_issue：修复迭代参数
  • regression：回归测试套件
  • 写 hooks（第二重要）
  • 每个 role 的 start / stop / logs 脚本
  • 每个环境的 prepare_env / clean_env 脚本
  • 遵循 hook 协议（source hook-helpers.sh + pg_exit）
  • pg doctor 校验
  • 跑一个最小 /2b-pg-quick-build 验证全链路通
关键提醒：project.yaml 的 modules.<m>.build 字段可以直接指向 .pg/hooks/ 下的脚本——比如 backend-build.sh 虽然是一个脚本文件，但它不在 hook 协议中，而是被 runner 当作普通 shell 命令执行。不要被"脚本在 hooks 目录下"误导，是否走 hook 协议取决于它在 project.yaml 的哪个位置。

4. 日常开发工作流

本章结构：4.1 全景路线图 → 4.2 定界 → 4.3 提案 → 4.4 构建。每节都讲核心机制 + 实操要点。

4.1 概述 —— 全景路线图

输入/输出对照表

阶段入口输入输出

4.2 定界 —— `/1-pg-define`

"接到需求先别急着写代码，问自己三个问题。"

第一个问题：需求有多复杂？

复杂度特征建议
过大 跨 ≥2 模块、涉 DB schema / K8s 变更、tasks 数量远超上下文窗口 拆分为多个 change，每阶段只做一件事，分周推进
适中 单模块、≤8 tasks、无 DB schema 变更、有明确边界 走标准流 /2-pg-propose + /3-pg-build
过小 改按钮文案、修 CSS、调整提示文字 直接 vibe coding，完全没必要走 pg-* 流程

关于"拆"的建议

比如"优化创建 VM 的性能" → 拆成 (1) 模板磁盘预缓存、(2) 异步状态推送、(3) 前端进度条展示，三个 change 分三周推进。每个 change 独立 propose → build → merge，互不阻塞，且每个都能在 LLM 上下文窗口内完成。

第二个问题：正经功能还是临时修 bug？

正经功能 → 标准流

/1-pg-define → /2-pg-propose → review → /3-pg-build

Bug → 修复流

/5-pg-fix-issue（系统化诊断 + 修复，详见第 5 章）

第三个问题：scope 想清楚了吗？

/1-pg-define 是一个探索姿态，AI 不是机械提问而是思考伙伴。它会帮你做这些事：

  1. 澄清模糊表述 —— "太慢是多慢？具体哪个场景？"把模糊词转化为可测量描述
  2. 划定边界 —— 影响范围是单一模块还是跨多个组件？明确 in / out of scope
  3. 暴露隐式假设 —— 挑战"必须用 X"的前提，打开替代方案空间
  4. 识别集成点 —— 会触及哪些现有功能？依赖其他系统或团队吗？
  5. 对标既有模式 —— grep 同类功能，遵循项目既有风格
  6. 揭示风险 —— 破坏性变更？兼容性？边界情况（并发、权限、空值）？
  7. 收敛完成标准 —— 怎么算做好了？区分 MVP 和迭代

⚠ 探索中的禁忌（自检信号）

  • 🔴 过早解决：跳过对问题的理解就"直接用 X 就行"
  • 🔴 功能蔓延：用户每说一个想法就延伸 3 个额外功能 —— 记住 YAGNI
  • 🔴 过度工程化：设计一个通用框架处理所有场景
  • 🔴 确认偏误：只在代码库中找支持自己假设的证据
  • 🔴 写代码冲动：发现自己在想"那行代码怎么写"而非"方案是否合理"

定界完成后的三个选择

边界清晰后，AI 会用 question 工具给你三个选项（用户做选择，AI 不替你决定）：

选项适用
走 /2-pg-propose正经功能、跨模块、需要设计评审
走 /2b-pg-quick-build明确小改动、≤8 tasks、单 module
直接 vibe coding改文案/样式，无需任何 SKILL

4.3 提案 —— `/2-pg-propose`

"边界有了，现在出图纸。图纸有四份，每份给不同的人看。"

四份文档 + 一份 SSOT

文件读者职责
proposal.md产品 / 架构师做什么、为什么、不做哪些
design.md开发者怎么做、架构决策、V-* 验证标准
tasks.mdPipeline Runnerstages × tracks 任务矩阵
1-propose-review/review-notes.md人机评审6 类自审项 + checkbox 追踪
environment.yamlRunner 部署引擎每个 stage 选哪个 environment（SSOT）

核心机制：tasks.md 的 stages × tracks 展开

pg-propose 读取 project.yaml 的 stages 段后，逐 stage 做两步过滤：

Step 1：on_conditions 判定

检查本 change 的 affected_paths 是否命中该 stage 的 on_conditions 规则：

  • 未命中 → 该 stage 整体跳过，tasks.md 中不生成任何章节
  • 命中 → 进入 Step 2

Step 2：affected_tracks 判定

对 stage.tracks 列表中的每个 track：

  • 在 affected_tracks 中 → 展开为 T→D→V→G 四个子章节（含具体任务）
  • 不在 affected_tracks 中 → 生成占位章节 - 无

environment 选择：影响 runner 的部署行为

对于 environment.required: true 的 stage，pg-propose 会读取该 stage 的 selection_rules，根据本 change 的 affected_tracks 选定 environment 并写入 environment.yaml。

场景选择依据
只改 backend + frontenddev-local单机能跑，无需跨主机
改 agent 且需要双 agent 通信dev-3tier跨主机拓扑，验证 libvirt 连接
默认dev-local评审阶段可升级到 dev-3tier

⚠ environment 选错代价很大

  • runner 在错误的 host 上启服务 → 启动失败
  • 用错的 hook 脚本 → 端口冲突或环境变量缺失
  • 跨机通信走错网段 → 联调失败

environment.yaml 缺失 → workflow_failed 硬性终止。这是 SSOT 的硬约束：runner 不知道该用哪个环境，宁可终止也不猜。

评审闭环：review-notes.md + /2.1-pg-propose-refine

  1. pg-propose 自审生成 review-notes.md（6 类审查项 + checkbox）
  2. 人工评审，把发现的问题在 review-notes 中标 [x] 或 [~]
  3. 执行 /2.1-pg-propose-refine，AI 按 scope 精准修改 proposal / design / tasks
  4. 通常需要 1-2 轮 refine 迭代，直到 review-notes 全部 [x]

人工 review 的硬规则：API 契约变更、DB schema 变更、跨模块能力演进，必须经人工 review 后才能进入 build 阶段。

4.4 构建 —— `/3-pg-build`

"图纸审完了，Runner 开始施工。人只做一件事：执行命令等结果。"

驱动流程：调用-派送-记录 循环

Runner 自动管的事（你不用操心）

  • 切到 feat/pg/<change> feature branch
  • 启动时 init commit（chore(<change>): bootstrap pg-build）
  • 每个 sub 阶段 record commit（chore(<change>): auto-record <item>:<sub> <status>）
  • 报告顺序编号 {track.id}-{N}-{kind}.md
  • final-gate pass 后自动归档到 changes/archive/

TDVG 四阶段：每个 track 内部循环

环境准备与清理

当 stage 的 environment.required: true 时，runner 在进入该 stage 前自动执行 prepare_env，完成后自动 clean_env。

使用哪个环境？由 environment.yaml 中该 stage 的字段决定（pg-propose 阶段已选定）。runner 通过 pg-invoke-hook.py invoke-hook --action prepare_env 触发，不传 --role / --instance（env-level 操作）。

幂等性：prepare_env 是幂等的——多次执行不会产生副作用。hook 脚本内已做"已就绪则跳过"的判断。

流程中断怎么恢复

中断场景恢复方式
会话中断（网络/超时） 重新 $RUNNER next，runner 从断点恢复，不丢失进度
Sub-agent 返回空 $RUNNER record failed 触发重试（最多 max_fail_retries 次）
workflow_failed（铁律） 人工介入：修复根因 → 删除 2-build/.pipeline-state.json → 重新 $RUNNER next → runner 扫描 tasks.md checkbox，从第一个含未完成任务 [ ] 的章节恢复

⚠ workflow_failed 恢复的关键点

不是从头开始！runner 重新扫描 tasks.md，定位到第一个未勾选 [ ] 的章节作为恢复起点。已完成的 [x] 章节全部跳过。这就是为什么 propose 阶段每个子任务都必须有 checkbox —— 它是 build 阶段断点恢复的依据。

副产物目录结构

.pg/changes/<change>/
├── proposal.md                       ← pg-propose 产物（不动）
├── design.md                         ← pg-propose 产物（不动）
├── tasks.md                          ← pg-propose 产物（runner 持续更新 checkbox）
├── environment.yaml                  ← pg-propose 产物（per-change SSOT）
├── 1-propose-review/
│   └── review-notes.md               ← 评审历史（归档保留，build 不读）
└── 2-build/                          ← pg-build 过程产物
    ├── context-chain.md

8. 常见问题与排查

Q&A

Build 跑到一半失败了怎么办？

查看 .pg/changes/<name>/2-build/*.md 中的报告。gate 或 verify 报告会写明失败原因。修复后重跑 /3-pg-build。

怎么知道当前 pipeline 跑到哪一步了？

查看 .pg/changes/<name>/2-build/.pipeline-state.json。

提案评审时发现了问题，怎么改？

在 review-notes.md 中标记 [ ] 为 [x]（已修）或 [~]（跳过），然后执行 /2.1-pg-propose-refine。AI 会自动按 scope 精准修改 proposal / design / tasks。

怎么新增一个回归测试套件？

编辑 .pg/project.yaml 的 regression.suite 段，增加一个新 suite 配置：

regression:
  suite:
    my-new-suite:
      environment:
        name: dev-local
        required_roles: []
      module: backend
      test_keys: [unit]
怎么加一个测试环境（比如 staging）？

编辑 .pg/project.yaml 的 environments 段，新增一个环境 + 各 role 的 actions。然后为该环境编写对应的 hook 脚本（prepare_env / clean_env / role start/stop/logs）。

pg-skills 版本怎么升级？
# 升级到最新版
pg upgrade

# 升级到指定版
pg upgrade v0.3.0

# 查看远程可用版本
pg upgrade --list

# 校验
pg doctor
怎么验证 hooks 写得对不对？

用 pg-invoke-hook.py 的 logs 动作（只读操作，安全无害）验证 hook 脚本是否能正常加载并执行：

python3 .pg/skills/src/runtime/bin/pg-invoke-hook.py invoke-hook \
  --change test-hooks \
  --env dev-local \
  --role backend \
  --instance backend-1 \
  --action logs \
  --tail-lines 10
module 命令和 hook 脚本的区别到底是什么？

核心区别两点：

  1. 是否有 PG_* 注入：hook 脚本有 $PG_ROLE、$PG_ENV 等；module 命令没有
  2. 错误处理方式：hook 脚本用 pg_exit / pg_fail 写 result.json；module 命令只看 exit code

判断标准很简单：看它在 project.yaml 中配置在哪个位置——modules 下的都是 module 命令，environments 下的都是 hook。
