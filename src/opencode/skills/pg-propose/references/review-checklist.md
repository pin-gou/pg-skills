# Review Checklist

本文档定义 pg-propose 阶段三自审时使用的 6 类系统化检查清单。
LLM 在生成 `1-propose-review/review-notes.md` 之前，必须对照本文档对 proposal / design / tasks 进行系统化检查。

> **写入规则与 review-notes 格式**：见 [./review-notes-format.md](./review-notes-format.md)

---

## 3.5.1 范围一致性（proposal vs tasks）

| 检查项 | 说明 |
|--------|------|
| 范围遗漏 | proposal "包含" 中说要做，但 tasks 找不到对应实现步骤 |
| 范围蔓延 | tasks 做了 proposal "不包含" 明确排期外的事情 |
| 范围模糊 | proposal 范围描述太宽泛，无法判断某任务是否属于该变更 |

---

## 3.5.2 API 完整性（design）

逐个端点检查：

| 检查项 | 说明 |
|--------|------|
| 请求体完整 | 每个字段是否标注类型？必填/可选？ |
| 响应完整 | 成功响应、错误响应是否都定义？ |
| 状态码 | 是否标注 200/201/400/404/500？ |
| 权限 | 是否需要鉴权？scope 是什么（平台/租户/项目）？ |
| 边界情况 | 参数为空、不存在的 ID、重复创建等是否有说明？ |

---

## 3.5.3 设计缺陷（design）

| 检查项 | 说明 |
|--------|------|
| 数据模型 | 字段类型、长度、默认值、唯一约束是否明确？ |
| 异常处理 | 业务异常时返回什么？调用方如何区分"连接失败"和"参数错误"？ |
| 安全 | 凭证是否明文传输？是否有加密/脱敏？ |
| 性能 | 是否存在 N+1 查询、大表全扫描等隐患？ |
| 幂等 | 重复调用是否安全？ |

---

## 3.5.4 任务歧义（tasks）

逐条检查每个 `- [ ]` 项：

| 检查项 | 说明 |
|--------|------|
| 动作明确 | "实现功能"太模糊——具体实现什么？哪个文件？ |
| 上下文完整 | 执行者是否知道从哪里开始、依赖什么前提？ |
| 验收标准 | 完成后怎么验证做对了？ |
| 依赖顺序 | 任务之间的依赖关系是否正确？特别注意：**verify 任务不能依赖后续阶段/agent 的产出**——例如 backend:dev-mock-integration 的 verify 任务不应依赖 frontend:dev-isolated 完成后才能验证，否则编排器按 stages 顺序执行时会导致 backend verify 卡住 |
| 文件路径 | 是否标注了需要修改的具体文件路径？ |

---

## 3.5.5 验证流程（tasks + design）

比对 `design.md` 的 Verification Criteria 和 `tasks.md` 的 verify 阶段任务：

| 检查项 | 说明 |
|--------|------|
| 覆盖率 | 每条 V-* 是否对应了 verify 阶段的具体任务？ |
| 可测试性 | "系统应该稳定"无法测试——是否替换为具体指标？ |
| 负面场景 | 是否覆盖了错误路径？（参数非法、资源不存在、服务不可达等） |
| 测试数据 | 验证时需要的测试数据是否有说明或准备步骤？ |
| 跨 stage 依赖 | verify 任务是否依赖后续 stage 的产物？每个 track × stage 的 verify 只应验证本 stage 范围内的工作 |

---

## 3.5.6 测试案例影响（tasks）

| 检查项 | 说明 |
|--------|------|
| 受影响测试 | 变更是否破坏已有测试？需标注哪些已有测试需要同步修改 |
| 新增测试覆盖 | 新功能是否有对应测试？是否在 tasks 中明确标注？ |
| 测试数据 | 测试所需的测试数据是否已准备或有准备步骤？ |
| 测试隔离 | 测试之间是否有依赖/执行顺序要求？ |

---

## 3.5.7 runner 可消费性（tasks）

| 检查项 | 说明 |
|--------|------|
| 验证记录 | 1-propose-review/ 下 validate-tasks-{round}.md 最后一份的 valid 是否为 true？ |
| 超限残留 | 若 2e.5 循环超限，残留的 error 是否已记录到 review-notes 的"阻塞"段？ |

---

## 写入 review-notes.md

把 6 项检查的发现 + 5 项通用决策合并写入 `.pg/changes/<change-name>/1-propose-review/review-notes.md`。
详细格式与编辑指引见 [./review-notes-format.md](./review-notes-format.md)。

---

## 阶段三行为契约

- **禁止**使用 `question` tool 中断流程
- **禁止**自动修改 proposal/design/tasks 主体内容
- **唯一允许的产物修改**：纯格式问题（如 markdown 标题层级错乱、代码块语言标记缺失、明显笔误），且修改后必须在 review-notes.md 中留痕记录"格式修正: X→Y"
- 自审完成后更新 TodoWrite 第 7 项为完成
