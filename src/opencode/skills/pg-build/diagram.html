判定条件与术语说明（基于实际代码）

v1 / v2 双状态机：由 _use_state_v2() 决定——环境变量 PG_USE_STATE_V2=true 或 project.yaml state_v2.enabled=true 时走 v2。v1 用 tasks.md checkbox 推断状态；v2 用 PipelineState（state.json 为唯一 SSOT，next_pending() 驱动 dispatch）。

Pipeline 顺序：由 get_pipeline_order() 从 execution-manifest.yaml staging 或 config.yaml stages[].tracks 生成。当前配置只有 dev stage（tracks: backend → agent → openapi-gen → frontend 顺序执行）。每个 track 内按 test → dev → verify → gate 依次推进。

v2 prepare_env 内联：v2 路径下 pg_build_bootstrap 的第 5 步同步执行 execute_env_hook_inline("prepare_env")，失败时直接返回 env_hook_failed action（不再走 v1 的 phase_result → LLM record 路径）。

v1 prepare_env/clean_env：v1 路径仍通过 _execute_phase 执行，返回 phase_result → LLM 调用 record completed 推进。

记录号体系：所有子 agent 报告文件（2-build/{track}-{N}-verify.md 等）由 runner 预分配全局递增 3 位 seq（dispatch_seq / report_seq）。SEQ 从 dispatch_history 自增，与 manifest 中 execution-seq 对应。

Gate-Fix 耗尽：当 gate-fix 循环达到 max_gate_fix_retries（默认 2）仍未修复时，未修复的 gap 追加到 known-issues.md，track 标记为 completed（接受 gap 继续推进，不阻塞 pipeline）。

Fix 循环耗尽：当 verify-fix 循环达到 MAX_FIX_CYCLES = 4 时，强制进入 gate 阶段（不再继续 fix）。此时 gate 仍然可以 pass/fail。

Record 状态守卫：LLM 用错 record status（如 verify 完成后调 record pass）返回 action: error, fatal: false 并附带 fix_hint，不污染 state。各 sub 允许的 status 集见图中虚线标注。

Auto-commit：每次 record 后 runner 同步执行 git add -A + git commit，message 格式 chore({change}): auto-record {item}:{sub} {status}。不 push。init commit 格式 chore({change}): bootstrap pg-build。

Final Gate + 归档：所有 track 完成后进入 final-gate，agent 检查跨 track 依赖项（design.md 🆕 标记）→ PASS 则 _auto_archive 将 change 目录移到 archive/YYYY-MM-DD-{change}/ + git commit → done。
